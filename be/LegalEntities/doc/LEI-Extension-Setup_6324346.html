R&D : LEI Extension Setup

Created by karol.szczepanski, last modified by robert.trypuz on Jul 31, 2015

LEI ontology (LEO) is intended to represent a domain of beings described by the content of LEI data file (specified in LEI Data File Format 1.0: https://www.gleif.org/content/4-events-and-media/1-press-releases/7-lei-data-file-format-1-0/lou-20140620.pdf and XML representation: https://www.gleif.org/_documents/lei-data-file-format.xsd).

It is decided that the notations used in LEO and their contents should be as close as possible to GLEIF's representation. Of course we should keep in mind that GLEIF's work specifies primally file format and secondary the legal entities and their properties. That's why some of the GLEIF's specification elements shall be omitted or reinterpreted in order to access to their meaning. For instance the elements Line1-Line4 (in table bellow) are to be primarily represented as FIBO's "PostalAddress" class and then further specified.

The following convention was agreed:

  • to use camelCase notation for names of Data- and ObjectProperties and PascalCase for Classes and Individuals
  • to add selected skos and FIBO annotations to the classes, properties and individuals of LEO (see the list below)
  • to add OWL axioms (mostly with cardinality restrictions) everywhere it is possible to better convey the meaning
  • to restrict DataProperties range by standard XSD datatypes
